什么是 LFS

LFS（Linux From Scratch）不是装现成发行版，而是从工具链到用户态组件逐层构建系统。 你会真正理解“为什么能启动、为什么会失败、为什么路径污染会造成连锁故障”。

  • 从 binutils/GCC 到 glibc 的依赖关系
  • 从内核到 init/systemd 的启动链路
  • 从构建错误到运行故障的系统级排障方法

为什么是 RISC-V

  • 启动路径清晰：OpenSBI -> Kernel -> init
  • Host/Target 边界明确，强制建立正确交叉构建思维
  • 方法可迁移：学会一套，可迁到其他架构与场景

`RISC-V-From-Scratch` 不只是“把系统做出来”，还要把你的工程判断讲清楚。 先完成主线，再展示你的优化、自动化、加固或可观测性设计。

三元目录隔离（铁律）

任何偏离此结构的构建都将引入路径污染与权限混乱

${CLFS}/

最终目标系统根目录（riscv64 架构）

cross-tools/

Host 架构可执行文件（x86_64→riscv64 交叉编译器）

tools/

Target 架构可执行文件（riscv64，通过 QEMU 解释执行）

sources/

源码与构建目录（普通用户权限操作）

关键原则

  • • cross-tools 与 tools 绝对分离
  • • 所有构建过程不得在 ${CLFS} 外安装任何文件

版本锁定矩阵

RISC-V 生态演进快速，必须预先验证以下兼容性

组件 推荐版本 注意事项
QEMU ≥ 8.0 低版本无法正确处理 glibc 2.38+ 的 clone3
GCC 13.x 与 Binutils 版本需匹配
Binutils 2.42 -
Linux Kernel 6.6.x 头文件版本不必与运行内核完全一致
systemd 随构建清单锁定 建议与当前 glibc/QEMU 组合完成兼容性验证

🏗️ 三段式构建核心方法论

Stage 1

构建交叉编译器

在 x86_64 宿主机上构建 riscv64-unknown-linux-gnu-gcc 工具链

Stage 2

构建临时系统

安装内核头文件、glibc，重建完整 GCC

Stage 3

构建最终系统

安装 coreutils、bash、systemd，完成主线启动配置

编译器构建三 Passes（CLFS 正统）

传承的交叉编译器构建流程

Pass 1

最小化交叉编译器（无 libc）

# Binutils（无依赖）
../binutils-2.42/configure \   --prefix=${CLFS}/cross-tools \   --target=${TARGET} \   --with-sysroot=${CLFS} \   --disable-nls --disable-werror

# GCC（仅 C 语言，无 libc）
../gcc-13.2.0/configure \   --prefix=${CLFS}/cross-tools \   --target=${TARGET} \   --with-sysroot=${CLFS} \   --enable-languages=c \   --disable-shared --disable-threads --without-headers
Pass 2

C 库安装

glibc 路径：先安装内核头文件，再构建 glibc，使用 Pass 1 GCC

Pass 3

完整交叉编译器

重新构建 GCC，启用 --enable-threads=posix 与 --enable-shared

⚠️ RISC-V 架构特化要点

特殊注意事项

页大小

RISC-V 支持 4KB/16KB/64KB 页大小，但用户态软件常硬编码 4KB 假设

libc_cv_pagesize=4096

启动流程

OpenSBI (M-mode)
↓
Linux Kernel (S-mode)
↓
init (U-mode)

无需构建 EDK2/UEFI

链接器检查

出现“文件存在但无法执行”时，优先检查 ELF 解释器路径与 glibc 链接器是否匹配

readelf -l /bin/ls | grep interpreter

压缩指令集 (RVC)

确保工具链与内核均启用 RVC 支持

--with-arch=rv64gc

❓ 常见问题

遇到问题？这里有答案

精选 20+ 个常见问题，基于历届试炼者的实际踩坑经验整理

原因：内核缺少 VirtIO 块设备驱动，或 root= 参数与设备名不匹配

# 确保内核配置包含：
CONFIG_VIRTIO_BLK=y
CONFIG_VIRTIO_PCI=y

# 检查 QEMU 参数，确保设备名匹配：
-append "root=/dev/vda rw"

原因：qemu-riscv64-static 未复制到 chroot 环境，或 binfmt_misc 未注册

# 1. 复制 QEMU 解释器
cp /usr/bin/qemu-riscv64-static ${CLFS}/usr/bin/

# 2. 注册 binfmt_misc
sudo update-binfmts --enable qemu-riscv64

原因：常见于 /etc/fstab、root= 参数或关键挂载点不一致

# 检查失败单元与启动错误
systemctl --failed
journalctl -b -p err

# 核对设备与 fstab
lsblk -f
cat /etc/fstab

原因：通常是 ELF 解释器路径与 rootfs 中链接器不匹配

# 排查解释器路径
readelf -l /bin/ls | grep interpreter

# 确认 glibc 链接器存在
ls ${CLFS}/lib/ld-linux-riscv64*.so.1

原因：glibc 2.42 需要 QEMU 支持 termios2，否则程序会崩溃

# 检查 QEMU 版本
qemu-system-riscv64 --version

# 解决方案：
# 1. 升级到 QEMU 10.1.2+
# 2. 或使用打补丁的版本
# 3. 临时降级 glibc 到 2.40

原因：没有安装可用 shell（推荐 bash）

# 先检查 shell 是否存在
test -x ${CLFS}/bin/bash || test -x ${CLFS}/bin/sh || echo "缺少 shell"

# 确保 /bin/sh 指向可用 shell
ln -sf bash ${CLFS}/bin/sh

原因：内核未启用串口控制台，或设备树不匹配

# 确保内核配置包含
CONFIG_SERIAL_8250=y
CONFIG_SERIAL_8250_CONSOLE=y
CONFIG_HVC_RISCV_SBI=y

# 启动参数添加 earlycon
-append "earlycon=sbi console=ttyS0"

原因：chroot 环境下 PID 1 通常不是 systemd，这属于预期行为

# 在正式启动后的目标系统中验证
ps -p 1 -o comm=
systemctl status
systemctl list-units --type=service --state=running | head

原因：安装脚本尝试修改文件组，某些环境下权限不足

# 检查必要工具是否已安装
ls -l ${CLFS}/bin/mount
ls -l ${CLFS}/sbin/agetty

# 如果都存在，可忽略 chgrp 错误

原因：Stage 2 构建时未启用必要的库

# 配置时确保启用
../configure \   --enable-languages=c,c++ \   --enable-shared \   --enable-threads=posix \   ...

解决方案：配置网络设备和驱动

# 1. QEMU 启动参数
-netdev user,id=net0,hostfwd=tcp::2222-:22 \ -device virtio-net-device,netdev=net0

# 2. 客户端配置
ip link set dev eth0 up
dhcpcd eth0
echo "nameserver 8.8.8.8" > /etc/resolv.conf

原因：缺少 CA 根证书

# 编译安装 make-ca 及其依赖
# 然后更新证书
make-ca -g

# 参考 BLFS: https://linuxfromscratch.org/blfs/view/systemd/postlfs/make-ca.html

原因：新 root 不是挂载点，或旧 root 有繁忙的挂载点

# 正确流程：确保新 root 是挂载点
mount --bind ${NEW_ROOT} ${NEW_ROOT}
pivot_root ${NEW_ROOT} ${NEW_ROOT}/mnt

# 或使用 switch_root（initramfs 工具集提供）
exec switch_root /newroot /sbin/init

原因：在裸串口控制台上运行时，shell 无法进行作业控制

# 这是预期行为，可以忽略

# 如需消除警告，确保
ls -la ${CLFS}/dev/console # 应该有 c 5 1
exec /bin/sh -l # 使用登录 shell

原因：config.guess/config.sub 太旧，或 --build/--host 参数填反

# 更新 config.guess/config.sub
# 确保 --build 是宿主机架构
./configure \   --build="$(gcc -dumpmachine)" \   --host=riscv64-unknown-linux-gnu \   ...

🐛 常见故障速查

快速问题定位表

对照症状，快速找到解决方案

故障现象 根本原因 修复方法
VFS: unable to mount root 缺少 CONFIG_VIRTIO_BLK dmesg | grep virtio
Exec format error qemu-static 未复制或 binfmt 未注册 验证复制和注册步骤
not found (文件存在) 动态链接器路径错误 readelf -l | grep interpreter
内核启动后崩溃 设备树不匹配或缺少串口配置 添加 earlycon=sbi
systemd 进入 emergency mode `root=` 或 `/etc/fstab` 不匹配 检查块设备、fstab 与 failed units
chroot 黑屏 未安装 bash chroot 前必须先安装 bash
